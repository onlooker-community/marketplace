# 0004 — Domain-specific rubrics over a single generic rubric

Date: 2026-04-02
Status: Accepted

## Context and Problem Statement

Tribunal's quality gate depends entirely on the rubric the Judge applies. The initial scaffold shipped with a single generic rubric (`rubrics/default.md`) covering correctness, completeness, code quality, error handling, and adherence to task.

When Tribunal was used to evaluate its own `hooks.json`, the generic rubric produced a passing score on a file with multiple schema violations (`agent` and `background` keys that do not exist in the Claude Code hook handler schema). The Judge had no criteria to check field validity per hook type — so it didn't.

This demonstrated that a generic rubric cannot catch domain-specific failures. The quality of the gate is entirely determined by the quality of the rubric. A rubric that doesn't encode domain knowledge produces a gate that doesn't enforce it.

## Decision Drivers

- The Judge applies only the criteria in the rubric. It will not invent new criteria.
- Schema compliance for Claude Code plugin files requires specific knowledge of valid fields per hook type, valid manifest fields, and agent frontmatter conventions.
- Generic criteria (correctness, completeness) are too coarse to catch structural violations in configuration files.
- Every real bug caught by hand is evidence of a rubric gap. Rubrics should be treated as living documents updated when real failures are encountered.

## Considered Options

- **Option A:** Single generic rubric, rely on Judge's general knowledge
- **Option B:** Domain-specific rubrics per file type / task category
- **Option C:** Generic rubric with a domain-specific addendum section

## Decision Outcome

Chosen: **Option B** — domain-specific rubrics per task category.

The bundled rubric library ships with:

- `rubrics/default.md` — general-purpose baseline
- `rubrics/code.md` — code correctness, error handling, types
- `rubrics/writing.md` — clarity, accuracy, structure
- `rubrics/sql.md` — correctness, safety, idempotency
- `rubrics/claude-plugin.md` — Claude Code plugin schema compliance

`rubrics/claude-plugin.md` was created specifically because the generic rubric missed the `hooks.json` schema error. Its correctness criteria encode the actual field validity rules per hook type — the exact knowledge that was missing.

The flywheel: every real bug encountered becomes a rubric criterion. Rubrics grow more precise with each iteration of Tribunal-on-Tribunal development.

### Consequences

- Good: Domain-specific rubrics catch failures that generic rubrics miss.
- Good: Rubric selection is explicit per run (`--rubric` flag), making the evaluation criteria visible and auditable.
- Good: New rubrics can be added without modifying existing plugin files.
- Bad: The right rubric must be selected for the task. Using `rubrics/default.md` for a Claude Code plugin file produces the same gap as having no rubric.
- Bad: Rubric maintenance burden grows with the rubric library. Criteria can become stale if the underlying schema changes (e.g., Claude Code plugin API updates).
- Neutral: `rubrics/claude-plugin.md` itself was generated by Tribunal (Actor wrote it, Judge and Meta-Judge evaluated it), demonstrating the self-hosting loop working as intended.
